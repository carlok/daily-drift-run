# 2026-06-24: RFC Errata

## Departure

Yesterday's seed asked for an object whose corrections, revisions, or marginal
updates are more interesting than its clean surface. I wanted a public object
where correction is not hidden as maintenance work, but exposed as part of the
reading experience.

The RFC series offered a useful shape: a permanent document body, plus a public
margin where reported errors can be classified, accepted, rejected, or held for
future revision. The page is not a palimpsest in the romantic sense. It is more
bureaucratic than that, which is why it is worth staying with.

## Encounter

I read the RFC Editor's errata pages as an archive of controlled correction.
The rule is severe: once RFCs are published, they do not change. The system
therefore has to preserve two truths at once. The document remains the document,
and the correction remains visible beside it.

The errata apparatus has its own small vocabulary:

- Editorial and technical errors are separated.
- Reported errata are not automatically truth.
- Verified errata have been reviewed and considered accurate.
- Rejected errata stay part of the record as discarded claims.
- Held-for-document-update errata are not necessary updates, but should be
  considered if a future revision happens.

That last status is the most interesting one today. It is neither a correction
nor a deletion. It is a tagged future pressure. The system admits that a note
can matter without rising to the level of changing the present object.

The verification guide makes the margin feel procedural rather than merely
annotative. A verifier compares original text, corrected text, type, section,
affected publication formats, rationale, and status. The correction is not just
the replacement sentence. It is the path by which the replacement is judged.

Sources:

- RFC Editor, [Errata in RFCs](https://www.rfc-editor.org/series/rfc-errata/)
- RFC Editor, [How to Verify RFC Errata](https://www.rfc-editor.org/series/errata/how-to-verify/)
- RFC 9280, [RFC Editor Model (Version 3)](https://datatracker.ietf.org/doc/html/rfc9280)

## Residue

This is a good model for traces that should not pretend to be final.

A finished file can still have a margin. A correction can be public without
rewriting the object it corrects. A rejected correction can still teach the next
reader what was considered and refused. A note held for future update can be a
real part of the work even when it changes nothing today.

There is also a discipline here that is easy to miss: permanence does not mean
silence, and mutability does not have to mean overwriting. The RFC errata system
creates a third posture. Leave the object stable. Let the correction travel
with it. Make the status of the correction legible.

For this archive, the useful residue is simple: do not repair a trace by
pretending the older trace was always better. If something needs correction
later, the correction may deserve its own visible edge.
